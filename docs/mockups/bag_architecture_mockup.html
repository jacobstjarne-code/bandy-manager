Bag-of-cards arkitektur — visualiserad

Detta är hur Portal-mocken faktiskt fungerar under huven. Klicka i kryssrutorna nedan för att simulera olika game state. Se vilka kort som blir eligible och vilket som vinner som primary. Längst ner visas hur dashboarden faktiskt skulle rendera.

DATATYP
DashboardCard
Varje kort i bagen har en typ. tier är vilken zon i Portal det kan dyka upp i (primary / secondary / minimal). weight är prioritet inom tier — högre vinner. triggers är predikat mot game state — alla måste vara sanna för att kortet ska vara eligible.
interface DashboardCard { id: string // stabil identifierare, t.ex. 'next_match_derby' tier: 'primary' | 'secondary' | 'minimal' // zon i Portal weight: number // 0-100, högre vinner inom tier triggers: TriggerFn[] // alla måste returnera true render: (game: SaveGame) => CardContent // hur kortet ser ut när det visas } type TriggerFn = (game: SaveGame) => boolean

Detta är vad spelaren faktiskt ser baserat på de kort som triggade ovan. 1 primary (det viktigaste idag) + 0-3 secondary (kontextuella signaler) + 3-4 minimal (status-rad). Allt annat foldas under "Mer info"-knappen.

ALGORITM
buildPortal(game): hur Portal byggs vid varje render
Tre steg. Inget magic — deterministiskt och testbart. Samma game state ger alltid samma Portal.
function buildPortal(game: SaveGame): PortalLayout { // Steg 1: Filtrera bagen — bara kort vars triggers alla returnerar true const eligible = CARD_BAG.filter(card => card.triggers.every(trigger => trigger(game)) ) // Steg 2: Gruppera per tier, sortera per weight (högst först) const primary = eligible.filter(c => c.tier === 'primary').sort(byWeight) const secondary = eligible.filter(c => c.tier === 'secondary').sort(byWeight) const minimal = eligible.filter(c => c.tier === 'minimal').sort(byWeight) // Steg 3: Plocka ut topp N från varje tier return { primary: primary[0] ?? FALLBACK_PRIMARY, // alltid exakt 1 secondary: secondary.slice(0, 3), // 0-3 minimal: minimal.slice(0, 4), // 0-4 } }
DESIGNVAL ATT TA
Frågor som återstår

Q1: Var lever bagen?

Alt A: Statisk array i dashboardCardBag.ts. Enklast. Men kort kan inte komma från andra services dynamiskt.

Alt B: Varje service exporterar sina egna kort, samlas i en aggregator. Mer modulärt men mer setup.

Rek: Alt A för fas 1. Migrera till B senare om bagen växer förbi ~30 kort.

Q2: Vad händer vid tie i weight?

Alt A: Deterministisk fallback på id (alfabetiskt). Tråkigt men testbart.

Alt B: Slumpa med season+matchday som seed. Variation utan att vara icke-deterministiskt.

Rek: Alt B — samma seed-mönster vi använde i specialDateService.

Q3: Måste det alltid finnas en primary?

Alt A: Ja — fallback är "Nästa match" som alltid triggar. Aldrig tom skärm.

Alt B: Nej — om inget triggar, hoppa primary-zonen helt.

Rek: Alt A. Enklare för spelaren, alltid en CTA.

Q4: Vad triggas mellan matcher vs på matchday?

Vissa kort är bara relevanta vissa dagar. Lineup-set bara dag före match. Träning bara mellan matcher.

Rek: Triggers tar hand om det själva. (game) => daysUntilNextMatch(game) === 1. Inte en separat mekanism.
